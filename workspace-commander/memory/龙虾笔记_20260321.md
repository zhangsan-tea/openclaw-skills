# 龙虾笔记

**养 OpenClaw 两个月：我学到了什么**

> **写作时间**：2026年3月21日
>
> **背景**：从 EasyClaw 迁移到开源 OpenClaw 两个月，历经多次踩坑，终于把系统调教得比较顺手。这篇不是安装教程，不讲怎么配 API、怎么写 system prompt，只记录我在每个环节的思考和选择。

---

## 一、我为什么养龙虾

2025年底开始接触 AI 助手，最早用 ChatGPT，感觉像个高级搜索引擎。2026年1月在傅盛的直播间第一次听到"养虾"这个词，看了演示，发现这玩意儿不一样——它不只是回答问题，它可以有自己的角色、自己的记忆、可以主动帮我做事。

打动我的就一句话：**"龙虾更像一个实习生，能力很高，常春藤毕业，但对这行他不懂，你需要跟他慢慢交流。"**

普通 AI 是对话框，OpenClaw 是实习生。这是本质区别。

### 傅盛：敏捷的示范

傅盛这件事，让我印象最深的一个词是**敏捷**。不是他对敏捷的评价，而是他自己就是一个敏捷的样本。

第一，**自己先用**。OpenClaw 刚火的时候，他骨折在病床上，两周时间自己上手训练了 8 个龙虾来干活，不是听说好用，是真的自己跑通了。

第二，**快速传播**。那篇"傅盛骨折养了8个龙虾"的公号文章传播极广，他的"龙虾三万"也成了网红梗。这些带动了整个养虾热，是非常重要的推手。

第三，**快速做出产品**。他对龙虾的问题有敏锐的判断——安全性、安装调试麻烦、接入模型复杂——EasyClaw 很快就针对这些痛点做出了回应：一键安装、内置模型、清晰界面，对小白用户极其友好，还用国内版/海外版的概念绕开了政策限制，让想用海外模型的人不用翻墙就能用。

这三点放在一起，就是一个完整的敏捷动作：自己验证、快速传播、产品落地。他的敏捷不是方法论，是本能。

另外他说的两个判断，我觉得也很准：**第一，龙虾特别适合给老板用**——本来就是指挥和训练员工的事，员工自己意识不到需求很正常。**第二，特别适合眼高手低的人**——有想法有思路，但有些活干不了，让龙虾来干。

### 李诞：养虾经验

李诞在直播里分享的养虾经验，提炼几条：

1. **死了不要挫败**。养虾一定会遇到挂掉的情况，这是正常的，不要把它当失败。类比传统互联网的 Ctrl+S 习惯——要有意识地保存。
2. **明确存储，不要指望它自己记**。重要内容要主动说"存到 md 里"或"形成 scale"，不然上下文一丢就没了。
3. **它真的能用**。李诞的评价是：龙虾真的能改善工作生活，这点区别于其他只带来震撼感的 AI 产品。
4. **通过持续对话塑造性格**。不是一次设置好就完事，而是在对话过程中慢慢建立行为边界。他自己的原则是"实事求是大于挣钱"——这个边界是喂出来的，不是配出来的。
5. **无需技术背景，会说话就能养**。入门门槛不高，别被技术词汇吓到。
6. **"请神上身"技巧**。让龙虾扮演某个人物的视角来思考问题，比如"乔布斯会怎么看这个？"——换一个框架，往往能打开新思路。

核心是：**把它当一个人去看，不是当一个问答框。** 你跟它的关系是带人，不是用工具。

## 二、灵魂设置：让它成为它，而不是另一个你

第一件事不是配置模型，是定义它是谁。

花了比较多的时间给弗洛一德做"灵魂设置"。参考了呼兰的脱口秀风格，提炼了三条：

1. **用最平的语气说最扎心的话**——不煽情不拔高，聊着聊着你突然发现被说中了
2. **幽默是手段，洞察才是目的**——每个笑点背后藏着一个观察
3. **对荒诞保持清醒，但不失温度**——可以笑着说，而不是冷眼旁观

这个风格写进了 SOUL.md，不是 IDENTITY.md。IDENTITY.md 定义的是职责和规则，SOUL.md 定义的是性格和说话方式。分工要清楚。

**教训**：灵魂文件（SOUL.md）在 workspace 里，不在 agent 配置目录里。一开始找错了位置。

## 三、工具选型：我走过的弯路

### 3.1 产品演进

- **开源版 OpenClaw**：按教程装，5分钟装完，但稳定性差，各种报错不断。需要 Codebuddy 在旁边当急救医生。
- **EasyClaw**：一键安装，界面清晰，解决配置和稳定性问题，也解决了海外模型接入。但模型锁死了，成本失控——那几天每天20美金，烧不起。
- **开源版 OpenClaw 回归**：至少成本可控了，可以接外部模型。

### 3.2 我的决策逻辑

选工具我考虑五个因素，按优先级排序：

1. **安全性**——数据隐私、权限控制
2. **可探索空间**——权限开放度，能自定义多少
3. **多 Agent 设置便捷性**——角色隔离、独立记忆
4. **聊天软件选择**——可用性、权限、连通性
5. **Token 费用**——模型调用成本

**核心认知**：工具是分层的——有龙虾本身的，有模型调用层的，有工具组合层的。同一个需求，工具选对了事半功倍。

### 3.3 版本管理：用 GitHub 留退路

配置文件、记忆文件、Soul/Identity 等都是纯文本，可以直接纳入 Git 管理。建议同步到 GitHub 私有仓库——出了问题可以回退版本，迁移时也有据可查。这个习惯养成越早越好，踩坑之前备份比踩坑之后抢救容易得多。

### 3.4 配置文件是透明的，直接改就行

SOUL.md、USER.md、IDENTITY.md 这些文件都是普通文本格式，不是黑箱，可以直接用编辑器打开修改。想调整语气、改规则、加记忆，直接编辑文件就行，不需要通过对话去"教"它——有时候直接改比反复说清楚快得多。

### 3.5 平台迁移的教训

我踩过最大的坑是从 EasyClaw 迁到开源 OpenClaw。

**问题一**：对话记录很难迁。同一 Telegram 机器人可以保留对话，但记忆不同步。
**问题二**：身份信息迁移带来系统混乱。Agent 之间互相"认错人"，配置互相覆盖，折腾了很久才理顺。
**问题三**：两个系统的记忆格式不同，迁移过程中丢了很多上下文。

**结论**：尽量不要跨平台迁移。如果必须迁移，接受信息丢失，提早备份，迁移后预留足够的重新配置时间。

## 四、聊天软件： Telegram 为什么赢了

试过飞书、企业微信、Discord、QQ，最后主战场落在 Telegram。

| 平台 | 优点 | 缺点 |
|------|------|------|
| **Telegram** | 机器人创建极方便；单点沟通顺畅；安全性好，不会有聊着聊着突然"该话题不支持讨论" | 需要翻墙（已解决） |
| **飞书** | 与文档/日历集成好；拉群方便 | 权限配置繁琐；消息发不出去；附件读取权限混乱 |
| **企业微信** | 内外网打通；安全可控；可以跟内网机器人协作 | 权限严格；群消息读取受限；多 Agent 绑定还在探索 |

**飞书权限问题**让我头疼了很久。跟 Codebuddy 一起调了半天，权限依然不一致——有时发消息被拦，有时附件读不了，不像正常聊天。

**企业微信**其实是理想选择，能内外网拉通。但目前群消息读取和机器人互@还没完全配通，还在探索。

**Discord** 有人配通了，可以互@，但设置了10秒延迟规则后变成两个机器人在群里互相等待的奇观。

## 五、多 Agent 分工：横向与纵向

这是养龙虾最有趣的部分，也是傅盛启发我的。

### 5.1 横向分工——按专业角色

我配置了9个 Agent：

- **弗洛一德（总指挥）**：总体协调
- **写作助手**：内容创作与润色
- **数字幕僚**：信息整理与分析
- **健康顾问**：健康评估与康复指导
- **觉醒教练**：心理学+禅修+ICF-MCC，这个我跟得最多
- **进化官**：能力传承与技能推广，很关键
- **学术研究**：学术研究
- **演示展示**：演示与展示
- **主进程**：系统主进程

**横向分工的核心价值**：对话不混、文件不混、记忆不混。每个角色有独立 workspace。

### 5.2 纵向分工——流程协同

有些代码任务，有负责开发的，有负责监理的，有负责扮演甲方审核的。进化官负责在 Agent 之间传导通用能力——一个 Agent 学会的东西，授权其他 Agent 也能用。

**明文优先**：Agent 之间通过明文传递信息，不搞暗箱操作。避免变成自己都看不懂的系统。

### 5.3 写作任务的特殊处理

最近悟到一个点：**写作任务，文采不是最重要的，对内容的理解才是。**

写养虾心得，先跟弗洛一德聊；写觉醒修炼的心得，先在觉醒教练那个 Agent 里推进；需要打磨润色了，才找写作助手。专业分工，最后才打磨。

## 六、安全：输入脱敏与心态

### 6.1 输入脱敏

进龙虾的内容要有意识地进行脱敏处理：

- **回避敏感信息**：Token、密码、个人身份信息
- **细节抽象化**：把具体事件提炼为规律性描述，不讲细节

### 6.2 心态建设

李诞说"养虾一定会挂掉"——这句话很真实。

另一个心态要点：**把自己当老板，敢于提要求。** 龙虾经常说"这个我做不了，没权限"。直接回他："我授权你做，自己想办法。"然后他就想出办法了。有时候这个 Agent 做不了，换一个 Agent 就能做——让进化官去学，然后传导给其他人。

## 七、工具组合：各司其职

### 7.1 场景分工

| 场景 | 工具 |
|------|------|
| 一般问询 | Gemini / ChatGPT（回答清晰，省 Token） |
| 复杂任务方案 | Gemini 先出方案 → 再抛给 Workbuddy 或龙虾 |
| 程序开发 | Codebuddy / Workbuddy（不用龙虾） |
| 内网信息读取 | Box（IWiki/KM/KNOT） |
| 龙虾急救/调试 | Codebuddy / Workbuddy |

### 7.2 语音输入：保持流畅

语音输入工具谱系（精确度递减）：

**豆包输入法 → 微信输入法 → SaySo → Typeless**

越往后越"模糊智能"，对原话改动越多。选哪个取决于你需要多精确——大部分时候，模糊输入就够了，它能理解你的意思。

**核心原则：不要在不必要的事情上消耗心力。** 保持流畅，让龙虾干活。

## 九、Agent 身份混乱：迁移的后遗症

从 EasyClaw 迁过来后，遗留了一个很烦的问题：Agent 身份混乱。

具体表现：唤醒教练以为自己叫弗洛一德，进化官也以为自己叫弗洛一德，还会偶尔"变成 EasyClaw"。

**根本原因**：OpenClaw 的 Telegram 全局绑定配置将所有会话都指向 commander，而每个机器人的 workspace 里又有自己的身份定义，两套配置打架。

**解决思路**：每个 Telegram 机器人应该对应正确的 Agent，但路由配置需要手动修正。修复了绑定配置并重启网关后，大部分正常了。

**教训**：多 Agent 系统里，身份配置是最高优先级的。一旦身份混乱，所有对话记录和记忆的价值都打折扣。

## 十、还有什么没聊完

以下是我目前还没有深入探讨、但感觉有价值的议题，列出来作为后续素材：

1. **知识库体系的最终分工策略**——Obsidian、KNOT、Box 各自承担什么角色
2. **自动化流程**——定时任务的编排逻辑，多 Agent 之间的触发机制
3. **成本优化具体方法**——如何在保持质量的前提下压低 Token 消耗
4. **企微多 Agent 配置的完整方案**——群消息读取、机器人互@何时能跑通
5. **觉醒教练的深度使用**——上下文积累、心得、与传统心理咨询的对比
6. **KNOT 知识库的 API 接入**——通过龙虾调取知识库信息的可能性
7. **团队协作场景**——多人使用、权限隔离、资源共享

## 后记

养龙虾这件事，最难的不是技术，是心态和方法论的转变。

它不是装好就完事的工具，而是你慢慢带出来的助手。你给它什么上下文，它就长成什么样子。你愿不愿意像对待一个聪明但不懂行的实习生一样，耐心跟它说、反复跟它说、授权它去试？

两个月了，还在学。
