## 一、为什么要养龙虾

### 一句话结论

常规 AI 更像一个入口或工具，龙虾更像一个人。

| 核心能力 | 具体表现 | 我的理解 |
|------|------|------|
| **电脑权限** | 浏览器、硬盘、网络、API | 不只是聊天，而是真的能接触电脑和外部工具，能读文件、调接口、调用网页能力 |
| **记忆系统** | 长期记忆、任务清单、上下文追踪 | 不是一次性问答，而是能逐步理解你、积累经验、延续任务 |
| **持续执行** | 定时任务、自动巡检、主动提醒 | 不用每次都等你重新下指令，可以持续跑、持续盯、持续提醒 |

龙虾不是一个更会聊天的大模型，而是一个**有电脑权限、有记忆系统、能持续执行的 Agent**，更像一个 24 小时替你干活的数字员工。

### OpenClaw 与常规 AI 的关键区别（备选对照表）

| 维度 | 常规 AI（ChatGPT 等） | OpenClaw 龙虾 |
|------|----------------------|---------------|
| 角色与记忆 | 无固定角色，无跨会话记忆 | 每个 Agent 有独立身份、SOUL、记忆 |
| 主动行为 | 等指令 | 可以主动搜集、定时任务、跨 Agent 协作 |
| 多 Agent | 单一天花板 | 多个专业化 Agent，分工协作 |
| 工具扩展 | 插件有限 | MCP 工具生态、可自定义技能 |
| 部署方式 | 纯云端 | 可本地部署，数据更可控 |
| 成本 | 按订阅或按量 | 可接入各种模型，成本更灵活 |

### 傅盛的两个关键判断

1. **龙虾特别适合给老板用**——本来就是指挥和训练员工的事
2. **特别适合眼高手低的人**——有想法有思路，但有些活干不了，让龙虾来干

### 李诞：养虾经验

1. **死了不要挫败**。养虾一定会遇到挂掉的情况，这是正常的，不要把它当失败。类比传统互联网的 Ctrl+S 习惯——要有意识地保存。
2. **明确存储，不要指望它自己记**。重要内容要主动说"存到 Memory 里"或"形成 skill"，不然上下文一丢就没了。
3. **它真的能用**。龙虾真的能改善工作生活，这点区别于其他只带来震撼感的 AI 产品。
4. **通过持续对话塑造性格**。不是一次设置好就完事，而是在对话过程中慢慢建立行为边界。他的原则是"实事求是大于挣钱"——这个边界是喂出来的，不是配出来的。
5. **无需技术背景，会说话就能养**。入门门槛不高，别被技术词汇吓到。
6. **"请神上身"技巧**。让龙虾扮演某个人的视角来思考问题，比如"乔布斯会怎么看这个？"——换一个框架，往往能打开新思路。

核心是：**把它当一个人去看，不是当一个问答框。**

### 基本驱动规律（AI 通用规则）

不管是传统 AI 产品还是 OpenClaw，有两个最基本的驱动力：

1. **需求驱动**——大量重复性工作、人力消耗大但缺乏创造性、传统方法搞不定的问题
2. **好奇驱动**——探索新可能，试探工具能力边界

判断一个场景是否适合用 AI，从这两点出发。

## 二、工具选型

> **总原则**：如果有省事的办法，就不要用复杂的办法——当然前提是你对它的安全性有一定的了解。技术的事要么用一键安装的产品省心，要么请技术团队支持。尽量不要自己在技术层面纠结太多，否则消耗的心力比省下来的钱还多。

### 2.1 产品演进路径

- **开源版 OpenClaw**：按教程装，5分钟装完，稳定性差，报错不断
- **EasyClaw**：一键安装，界面清晰，解决了配置/稳定性/海外模型接入问题。但模型锁死，成本失控——每天20美金，烧不起
- **开源版 OpenClaw 回归**：成本可控，可接外部模型

### 2.2 决策五因素（按优先级）

1. **安全性**——数据隐私、权限控制
2. **可探索空间**——权限开放度，能自定义多少
3. **多 Agent 设置便捷性**——角色隔离、独立记忆
4. **聊天软件选择**——可用性、权限、连通性
5. **Token 费用**——模型调用成本

**核心认知**：工具是分层的——有龙虾本身的，有模型调用层的，有工具组合层的。同一需求，工具选对了事半功倍。

### 2.3 平台迁移的教训

最大坑是从 EasyClaw 迁到开源 OpenClaw：

- **对话记录难迁**：同一 Telegram 机器人可保留对话，但记忆不同步
- **身份信息混乱**：Agent 之间互相"认错人"，配置互相覆盖，折腾很久才理顺
- **记忆格式不同**：迁移过程丢了很多上下文

**结论**：尽量不要跨平台迁移。必须迁的话，提早备份，迁移后预留足够的重新配置时间。

### 2.4 版本管理：留退路

- **内网环境**：通过内网 Git 仓库备份配置文件、记忆文件、Soul/Identity 等纯文本
- **外网/开发环境**：通过 GitHub 私有仓库管理，出问题可回退版本，迁移时有据可查

**进阶用法**：结合定时功能，每天提醒自己记录当日进展——"今天干了什么？学到了什么新技能？还有什么待办？"，作为每日日志的一部分。

版本管理习惯养成越早越好。踩坑之前备份，比踩坑之后抢救容易得多。

**电源常开设置**（详细方法略，请自行检索）：基于 Mac 系统电源配置；终端侧 `caffeinate` 命令。

### 2.5 配置文件是透明的，直接改就行

SOUL.md、USER.md、IDENTITY.md 这些文件都是普通文本格式，不是黑箱，可以直接用编辑器打开修改。想调整语气、改规则、加记忆，直接编辑文件就行，不需要通过对话去"教"它——有时候直接改比反复说清楚快得多。

### 2.6 Codebuddy / Workbuddy：急救医生

Codebuddy 和 Workbuddy 是龙虾最重要的外援工具：

- 龙虾出了奇怪问题，截图或报错发给 Codebuddy，它能直接改配置，从底层修复
- OpenClaw 挂了，Codebuddy 有底层权限，可以直接操作修复，包括重启模型
- Workbuddy 还有远程功能，通过企微等渠道远程操控——龙虾死了之后，手机上用 Workbuddy 远程救治，很实用
- 基于 Codebuddy，编程能力很强，对本地电脑操控能力也很强

### 2.7 多产品体验 vs. 集中精力

前期可以多体验几个产品，多了解不同工具的能力和边界。但后期建议**集中精力在一两个工具上**深挖，不要每样都蜻蜓点水。体验是探索，集中才是生产力。

### 2.10 Token 消耗优化与成本控制

**常用方法：**

- **`/new`**：开启全新对话，彻底清空上下文，最省 Token
- **`/reset`**：保留记忆，清空短期上下文，适合换话题时用
- **`/compact`**：压缩当前会话，保留关键信息丢弃多余细节
- **多 Agent 分工**：不同任务用不同 Agent，避免记忆污染
- **memory-search 替代长对话**：让 Agent 主动查记忆，而非堆砌上下文
- **复杂需求先用 ChatGPT/Gemini 聊几轮**，形成清晰方案再抛给龙虾
- **稍复杂程序开发直接用 Codebuddy/Workbuddy**，可视化效果更好，不用龙虾
- 养成习惯：新话题用 `/new` 或 `/reset`，重要信息明确说"存到 Memory"
- 定期让龙虾生成 Token 报告，查看消耗排名，找出异常点

**本地部署省钱方案**：

可部署本地模型（如 Qwen3.5 7B/8B），好处是数据安全（模型层面），免费。缺点是模型能力比满血版差很多，只建议做非常简单任务，如归纳每日总结、设置日程提醒。稍复杂的任务运行非常慢且效果差。

### 3.1 先工具，后配置

选好了用哪个产品，再来配置它的细节。逻辑才对。

### 3.2 关键文件一览

> 这些文件的价值在于把你的身份、习惯和相关要求**透明化**了——这是龙虾区别于传统 AI 对话框的重要创新。初始设置时系统会问你，也可以打开直接修改，也可以用口播方式要求它改某些地方。

## 五、聊天软件

试过飞书、企业微信、Discord、QQ，主战场落在 Telegram。

| 平台 | 优点 | 缺点 |
|------|------|------|
| **Telegram** | 机器人创建极方便；单点沟通顺畅；安全性好，不会有聊着聊着突然"该话题不支持讨论" | 需要梯子 |
| **飞书** | 与文档/日历集成好；拉群方便；有飞书妙记等一键部署工具 | 权限配置繁琐；消息发不出去；附件读权限混乱；一键配置代码/工具可简化部署（详细方法略，请自行检索） |
| **企业微信** | 内外网打通；安全可控；可与内网机器人协作 | 权限严格；群消息读取受限；多 Agent 绑定还在探索 |
| **Discord** | 支持私信和服务器频道与 OpenClaw 对话；频道隔离做得好 | 回复延迟较长（需优化流式输出）；国内访问相对稳定 |

飞书权限问题头疼了很久，不像正常聊天。**企业微信**是理想选择，能内外网拉通，群消息读取和机器人互@还没完全配通，继续探索中。

**待办与仍在探索中：**
- 企微群消息读取完全配通
- 企微多 Agent 绑定跑通
- 企微与内网机器人（OpenClaw 内网版 / Box）的内外网联动
- 飞书权限彻底调通（包括利用一键配置工具）

## 六、多 Agent 分工

### 6.1 多 Agent 的两种思路

**横向分工（按专业角色）**：不同专业的事交给不同的 Agent，对话不混、文件不混、记忆不混。

**纵向分工（按流程环节）**：同一个任务，有负责开发的、有负责监理的、有负责审核的。

**混在一起 vs 分开，各有利弊：**

| | 混在一起 | 分开 |
|---|----------|------|
| **好处** | 信息集中，容易上下文共享 | 职责清晰，不互相干扰 |
| **坏处** | 久了容易角色混淆，身份混乱 | 需要主动协调，跨 Agent 信息传递 |

目前倾向**分开为主**，但信息偶尔共享时用明文传递，避免黑箱。

### 6.2 横向分工现状

| Agent | 角色 |
|-------|------|
| 弗洛一德 | 总指挥 / 养虾助理（承接模糊类需求） |
| 写作助手 | 内容创作与润色；沉淀规则到企业文档，支持查看和调整，根据新信息自动迭代 |
| 数字幕僚 | 工作类信息的整理与分析 |
| 健康顾问 | 健康评估与康复指导 |
| 觉醒教练 | 心理学 + 正念禅修 + ICF-MCC 教练；帮助发现潜能、盲区、模式，探索更多可能 |
| 进化官 | 能力传承与技能推广；拥有最高权限，可读取和修改其他助手的设置 |
| 主进程 | 系统主进程 |

**Agent 使用要点：**

**觉醒教练：**

- **清晰的提示词和角色设定**：设定为有心理学和正念禅修背景的 ICF MCC 教练，帮助发现潜能、盲区、模式，探索更多可能。这个设定是基础，决定了对话的深度和方向。
- **足够的上下文和背景信息**：主动提供个人相关情况，包括个人经历、对话记录等，帮助它往深处思考，而不是停留在表面。
- **日常互动积累默契**：通过持续对话，慢慢觉得它很懂你，你也越来越知道怎么用它——它对你的情况、特点、模式、盲区以及变化会越来越敏锐。这个在内观类应用上非常明显。
- **与知识库结合**：复杂文档可以放到知识库里，利用龙虾的整体分析能力来看情况；也可以利用定时能力，定时拉取新信息（如来自 flomo 的随手记），跟踪新议题和情况的变化。

**健康顾问：**
- 输入体检报告、健康问卷等，建立个人健康档案
- 设置定时体检/检查任务、运动康复计划、服药提醒
- 日常健康问题互动，让其记住关键信息
- 与苹果健康打通，分析睡眠、心跳等指标（本周发现：睡眠与心跳某关联，待查）
- 定期扫描一周健康 APP 数据，给出本周发现和提醒

**进化官：**
- 整个系统的"能力传导器"，拥有最高权限
- 具体做三件事：**能力学习**（负责升级系统、学习新技能）、**技能传导**（一个 Agent 学会的通用能力向其他 Agent 推广）、**故障诊断**（其他 Agent 出问题后帮诊断和修复）
- **案例**：某 Agent 学到了处理某类文件的能力 → 进化官验证可通用 → 推广到所有需要的 Agent
- **明文优先**：Agent 之间通过明文传递信息，不搞暗箱操作，避免变成自己都看不懂的系统

**写作助手：**
- 文采不是最重要的，对内容的理解才是
- 写养虾心得，先跟总指挥聊；写觉醒修炼的心得，先在觉醒教练 Agent 里推进；需要打磨润色了，才找写作助手
- **专业分工，最后才打磨**
- 写作助手负责沉淀规则，这些规则存到企业文档中，可以随时查看和调整，并根据新信息进来自动迭代

**待办与仍在探索中：**
- 进化官能力传导机制的完善
- 纵向分工的流程规范化
- 多 Agent 之间自动触发和协作机制
- 各 Agent 使用要点的完整梳理

## 七、安全

### 7.1 敏感信息种类

以下信息类型需要特别注意脱敏或回避：

| 类别 | 示例 |
|------|------|
| **认证凭据** | Token、API Key、密码、私钥 |
| **个人身份信息** | 姓名、身份证号、护照号、驾照号 |
| **联系方式** | 手机号、邮箱、家庭住址 |
| **财务信息** | 银行卡号、支付账户、社保卡号 |
| **健康信息** | 病历号、处方、医保信息 |
| **工作敏感信息** | 内网 IP、内部系统账号、安全策略详情 |
| **可识别身份的信息** | 人脸、指纹、位置轨迹、行为习惯画像 |

**输入脱敏两条原则：**
- 回避上述敏感信息，不直接输入
- 把具体细节提炼为规律性描述，不讲细节

### 7.2 其他安全要点

- **模型选择**：优先选安全策略透明的产品
- **工具来源**：避免安装来源不明的 Skill/MCP
- **权限控制**：谨慎授权，不要过度开放
- **数据备份**：重要记忆和配置定期备份
- **区分身份**：个人身份与 AI 身份分开，避免泄露
- **涉敏任务**：优先使用 Workbuddy（闭源，权限隔离好）
- **大厂安全防护**：腾讯等大厂提供了安全检测类技能、更安全的技能仓库等；Windows 版还有电脑管家等防护能力（详细请自行检索）

## 九、知识管理

### 9.1 知识 vs 记忆

- **知识**：可复用、结构化、跨会话传递的公共信息
- **记忆**：个人化、情境化、随互动积累的私人信息

两者协同，但用途不同。知识用来回答问题，记忆用来理解"你是谁"。规律：知识是"知道"，记忆是"认识"。

### 9.2 输入格式

- 优先 **MD 格式**保存知识，结构清晰，易于检索
- 可通过 Skill 将 MD 转成讲述者模式（正在推敲中）
- 重要信息要**明确说"存到 Memory 里"或"形成 skill"**，养成习惯

### 9.3 知识的类型与权重

| 类型 | 特点 | 执行置信度 |
|------|------|-----------|
| 结构性/原理性信息 | 规律清晰，可直接指导行动 | 高 |
| 参考性资料 | 需要结合情境判断，不能直接套用 | 低，需标注来源 |
| 个人经验总结 | 因人而异，需要验证 | 中 |

### 9.4 知识库打通现状

| 平台 | 与龙虾的结合状态 | 说明 |
|------|-----------------|------|
| **Observe** | ✅ 已能很好结合 | 可直接调用 |
| **IMA** | 🔄 正在打通中 | 即将支持 |
| **NotebookLM** | ⚠️ 官方无对应方法 | 有民间 script，但不稳定 |
| **龙虾自带知识库** | ✅ 自身具备 | 复杂文档可存入 |
| **KNOT** | ✅ 打通中 | 内网知识库，可结合代码和 Tapd；与企微结合方便 |
| **IWiki / KM** | ✅ 内网文档检索 | 查内网资料方便 |
| **Box** | ✅ 读取+外网连通 | 非知识库定位，主要用于读取；可连通内外网；浏览器操控能力强 |
| **临时内容** | ✅ memory 或指定文件夹 | 临时性内容存在 memory 或特定文件夹 |

多平台打通是趋势，内外网信息协同是长期目标。

### 9.5 海量信息的批量处理

通过 Codebuddy / Workbuddy 读取本地文件夹：

- 批量去除重复文件
- 批量给文件重命名（文件名加创建时间）
- 给出文件分类建议
- 文字信息转 MD 格式（单文件和批量文件夹）
- PPT 视觉化理解与转述（这个需求非常大，正探索中）

**这类 Skill 需求量很大，正在探索中。**

## 十一、基本原则汇总

来自自身经验和傅盛/李诞直播的提炼：

1. **需求和好奇是最基本的驱动力**——不管是 AI 还是 OpenClaw，适合解决大量重复工作、人力消耗大但缺乏创造性、传统方法搞不定的问题
2. **工具是助手，人承担最终决策权**——责任不外包，这是基本原则；人对最终结果负责（经验、洞察、审美等），龙虾是工具
3. **意图比流程重要**——讲清楚要什么，而不是把自己的工作流强加给龙虾
4. **上下文是核心资产**——慢慢积累，让它越来越懂你；重要信息主动说"存到 Memory 里"
5. **敢于授权**——"我授权你，自己想办法"
6. **用好模型**——实习生聪明不聪明，差太多了；稍复杂任务，模型高一寸，工程胜一丈
7. **不要用自己的想法套住模型**——先丢给 AI 看规律，有新发现再叠加
8. **敢于回到第一性原理**——不要让自己的能力成为龙虾的天花板
9. **能明确表达需求**——多轮表达也没关系，这是人最重要的能力之一
10. **多互动，敢于用通俗语言反复聊**
